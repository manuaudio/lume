# Lume — Native macOS Rewrite (v1 Responsive Core)

**Date:** 2026-06-06
**Status:** Approved — building
**Supersedes:** the all-SPM prototype (preserved on an archive branch)

## Why

The SPM-prototype Lume became effectively unusable: clicks felt sluggish / didn't
register. Diagnosis (instrumented, 2026-06-06) showed the main thread was **idle**
during freezes, the window was **not frontmost**, and warm document switching was
~37ms — i.e. nothing was slow. The root cause was that Lume is a **bare SPM
executable** forced into GUI behavior via `NSApp.setActivationPolicy(.regular)` +
a single launch-time `NSApp.activate(...)`. A bare executable with no real bundle
gets unreliable activation and event routing.

Decision: **rebuild from scratch as a real, native macOS app.** Not "Lume 2",
not "1.0" — a fresh foundation for the product still called **Lume**. It must be
snappy, macOS-friendly, and work out of the box.

## Non-negotiables

- **Snappy.** File open is single-digit ms (native text, no WebView).
- **macOS-friendly.** Real app bundle, native window/activation, standard
  keyboard + menu behavior.
- **Works out of the box.** Launch the built `.app`, it opens a window, takes
  clicks immediately, lets you open a folder and edit files. No setup ritual.

## Scope — v1 "Responsive Core" (this spec)

In:
- Folder sidebar: open a folder, browse its file tree, select files.
- Native **TextKit 2** editor: open/edit/save text files, fast.
- Lightweight markdown syntax highlighting (headings, bold/italic, code, links).
- A responsive `NavigationSplitView` window that receives input immediately.
- Find (⌘F), undo, native scrolling.
- Non-text files: "Open in default app" / Quick Look affordance (no custom viewers yet).
- Remember last-opened folder + window frame across launches.

Out (later increments, each its own spec):
- Tags, GROUPS (tags-as-virtual-folders), pinning, display names, colorful tags.
- Custom viewers (PDF, image, code, env, config, HTML).
- File CRUD + Undo, drag-to-tag/pin, multi-select Copy Paths, hide/peek.
- App Sandbox, SwiftData persistence, Mac App Store distribution.

## Architecture

Real Xcode app project generated by **XcodeGen** (`project.yml` → `.xcodeproj`),
built with `xcodebuild`. Two targets:

### `LumeKit` (framework target — UI-free domain logic)
No AppKit/SwiftUI imports. Pure, unit-tested.
- `FileNode` — a node in the browsed folder tree (url, name, isDirectory, children lazy).
- `FolderScanner` — enumerates a directory into `FileNode`s (off-main, async). Skips
  hidden files by default; sorts directories-first then name.
- `FileKind` — classifies a URL: `.text`, `.markdown`, `.other` (by extension/UTType).
- `TextDocument` — loads file text (async, off-main), tracks dirty state, saves atomically.
- `MarkdownHighlighter` — tokenizes markdown into styled ranges (pure function over a
  String → `[HighlightToken]`); UI layer maps tokens to attributes.

### `Lume` (app target — thin SwiftUI/AppKit shell)
- `LumeApp` — `@main App`, `WindowGroup`, standard menus. NO manual activation hacks.
- `ContentView` — `NavigationSplitView`: sidebar (file tree) + detail (editor or placeholder).
- `SidebarView` — renders the `FileNode` tree, "Open Folder…" button, selection.
- `EditorView` — `NSViewRepresentable` wrapping an `NSTextView` on the **TextKit 2**
  stack; applies `MarkdownHighlighter` tokens; binds to `TextDocument`.
- `AppState` — `@Observable` view-model: current folder, selected file, open document.
  Bridges `LumeKit` to the UI. `@MainActor`.
- `Preferences` — thin `UserDefaults` wrapper: last folder (security-scoped bookmark),
  window frame is autosaved natively.

### Data flow
Open Folder → `FolderScanner` (off-main) → `[FileNode]` → sidebar.
Select file → `TextDocument.load` (off-main) → text + `MarkdownHighlighter` tokens →
`EditorView` renders. Edit → debounced re-highlight of changed region; save on ⌘S /
autosave-on-change.

## Persistence (v1)
- `UserDefaults`: last-opened folder bookmark, view prefs.
- One **security-scoped bookmark** to reopen the last folder.
- Window frame: native `setFrameAutosaveName` via SwiftUI scene.
- **No SwiftData in v1** — introduced with the tags/GROUPS increment.

## Sandboxing (v1)
- **Non-sandboxed, signed with local/Developer ID** to avoid bookmark friction during
  the rebuild. App Sandbox is turned on in a later distribution-focused increment.

## Concurrency
- Swift 6 language mode, strict concurrency.
- UI types `@MainActor`. File scanning + reading + saving are `async` off the main
  actor (the original bug's lesson: never touch disk/iCloud on the main thread).

## Error handling
- Folder open failure / unreadable file → inline, non-fatal banner in the detail pane;
  never crash, never block the window.
- Save failure → surfaced, document stays dirty.
- Binary/non-text selection → placeholder with "Open in default app" + Quick Look,
  not a parse attempt.

## Testing
- Swift Testing for `LumeKit`: `FolderScanner` (sorting, hidden-file skip, nesting),
  `FileKind` classification, `TextDocument` load/save/dirty, `MarkdownHighlighter`
  tokenization (headings, emphasis, code, links, edge cases).
- TDD: tests first for each `LumeKit` unit.
- The app target is verified by **actually building the `.app` and launching it** —
  window appears, takes clicks, opens a folder, edits a file. "Works before shipping"
  means observed, not asserted.

## Repo
- Keep the `lume` git repo (product identity preserved).
- Archive the SPM prototype on branch `archive/spm-prototype` before the rewrite lands.
- New project lives at the repo root: `project.yml`, `Sources/LumeKit/`,
  `Sources/Lume/`, `Tests/LumeKitTests/`. Old `Sources/`/`Package.swift` retired.

## Definition of done (v1)
1. `xcodebuild` produces `Lume.app` with a proper Info.plist + bundle ID.
2. Launching it shows a window that **immediately** takes clicks (the original bug gone).
3. Open Folder → browse tree → click a file → it opens in <50ms and edits smoothly.
4. Markdown files show syntax highlighting; ⌘S saves; ⌘F finds.
5. `LumeKit` tests pass.
6. Last folder + window position restored on relaunch.
